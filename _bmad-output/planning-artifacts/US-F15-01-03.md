# US-F15-01-03 — CDC Postgres → Neo4j (Supabase Realtime → Celery → Cypher)

**Épica**: EP-F15-01 (KG Foundation) | **Sprint**: S10 | **SP**: 8 | **Prioridad**: P0

## Contexto

Wave 3-A de S10 (paralelo con US-F15-02-01). Depende de US-F15-01-01.
El codebase ya tiene `CdcDispatcher` + `cdc_events` table + task `mt.graphrag.process_cdc_batch`.
Esta story extiende ese patrón para sincronizar `products` y `competitor_listings`
hacia Neo4j en tiempo real (< 5s de latencia) via Supabase Realtime → Celery.

## Descripción

Implementar pipeline CDC de baja latencia: Supabase Realtime detecta cambios en
`products` y `competitor_listings`, dispara webhook a un endpoint FastAPI que
encola la tarea Celery `sync_product_to_kg`, que ejecuta el Cypher de upsert en Neo4j.

## Criterios de Aceptación

- [ ] Nueva Celery task `mt.graphrag.sync_product_to_kg` en
      `app/workers/tasks/graphrag.py`:
      - Acepta `product_id: str`, `operation: str` (insert/update/delete)
      - Hace MERGE del producto en Neo4j via `get_default_graph_store()`
      - Retry exponencial: 3 intentos, delays 10s/30s/90s
      - Log estructurado con `product_id`, `operation`, `latency_ms`
- [ ] Modelo `app/schemas/cdc_event.py` (o extender existente) con campos:
      `table`, `operation`, `record_id`, `payload`, `received_at`
- [ ] Endpoint `POST /api/v1/internal/cdc/product` en nueva ruta
      `app/api/routes/internal_cdc.py`:
      - Verifica header `X-Internal-Secret` (settings.INTERNAL_CDC_SECRET)
      - Encola `sync_product_to_kg` via Celery
      - Returns 202 Accepted
- [ ] Supabase Realtime habilitado para tablas `products` y `competitor_listings`
      en `supabase/config.toml` (sección `[realtime]`)
- [ ] Propagación end-to-end < 5s (medible en tests de integración)
- [ ] Fallback manual: `process_cdc_batch` existente como respaldo si Realtime falla

## Notas Técnicas

- Supabase Realtime usa webhooks HTTP cuando se configura en `config.toml`
- La URL del webhook en dev: `http://api:8000/api/v1/internal/cdc/product`
  (nombre de servicio en docker-compose)
- `INTERNAL_CDC_SECRET` nueva var en `.env.example` y `app/core/config.py`
- El sync a Neo4j usa `GraphNode(label="Product", primary_key=product_id, ...)`
  y delega en `get_default_graph_store().merge_node()`
- Si `GRAPHRAG_BACKEND=stub` (default), la task es no-op (loguea y retorna)
- Tests de retry: mockear `merge_node` para lanzar excepción 2 veces
- Tests de idempotencia: llamar sync_product_to_kg 3 veces con mismo product_id
